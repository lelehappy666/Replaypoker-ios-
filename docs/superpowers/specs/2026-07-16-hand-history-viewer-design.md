# River Club 真实牌局存档查看器设计

日期：2026-07-16
状态：设计已确认

## 1. 目标

本子项目把侧边栏“我的牌局”从模拟牌桌列表替换为真实本地牌局存档。玩家可以随时查看已经完成并成功保存的普通桌牌局，查看九名玩家的最终底牌，包括已经弃牌的玩家。

本项目必须保持以下最高优先级边界：

- 只有已经完成并原子保存的牌局可以显示所有最终底牌。
- 进行中的牌局、正常牌桌界面、旁观观察和机器人决策不能通过存档功能取得对手底牌。
- 存档只展示最终结果，不提供逐动作牌谱或回放。
- 所有筹码均为无现金价值的本地娱乐筹码。

## 2. 本期范围

本期包含：

- 侧边栏“我的牌局”真实存档列表。
- 牌局最终结果详情。
- 按完成日期和牌桌组合筛选。
- 单局删除与全部清空，均需二次确认。
- 加载、空数据、筛选无结果、读取失败和删除失败状态。
- 结算后刷新、应用重启后的持久化恢复和横屏 Pro Max 自动化验收。
- 新存档冻结牌桌显示名和九个座位的玩家显示名；旧存档缺少名称时使用稳定降级文案。

本期不包含：

- 逐动作牌谱、动画回放或决策分析。
- 存档导出、分享、云同步或跨设备同步。
- 存档搜索、标签、收藏或无限滚动分页。
- 锦标赛存档。
- 完整离桌结算和离桌后重新入座。

## 3. 用户流程与页面结构

```text
侧边栏“我的牌局”
        ↓
真实存档列表（最新优先）
        ↓
点击某一局
        ↓
最终结果详情
```

### 3.1 存档列表

列表默认按 `endedAt` 从新到旧排列。每条记录显示：

- 冻结的牌桌显示名；旧记录没有显示名时显示 `牌桌 <tableID>`。
- 小盲/大盲。
- 完成日期和时间。
- 手牌编号。
- 真人本局筹码净变化，正数、负数和零值使用文字与符号共同表达。
- 本局已分配底池总额。
- 已发出的公共牌缩略图；弃牌提前结束时允许少于五张。
- “查看详情”入口。

列表不显示任何进行中牌局，也不从 `tableCoordinator.state` 推导历史记录。

### 3.2 最终结果详情

详情页显示：

- 牌桌显示名、盲注、完成时间、手牌编号。
- 该局全部公共牌；提前弃牌结束时按实际数量显示。
- 九个已入座座位的冻结玩家显示名、座位号、起始筹码、最终筹码和本局净变化。
- 每个实际获发底牌玩家的两张最终底牌，包括弃牌玩家。
- 座位最终状态：获胜、弃牌、摊牌或未参与。
- 主池和每个边池的金额、赢家及获得筹码。
- 未被跟注而退回的筹码。

详情页不显示逐步过牌、跟注、下注、加注、全下或弃牌时间线，不提供播放控制。

## 4. 导航

现有 `AppRoute.tables` 的侧边栏文案已经是“我的牌局”，本期保留路由标识以避免无关迁移，但把对应内容从 `TableListView` 替换为 `HandHistoryView`。

大厅“全部牌桌”不能再跳转到 `AppRoute.tables`，需要新增独立的牌桌浏览路由，例如 `AppRoute.tableBrowser`。该路由不加入侧边栏，仅供大厅进入牌桌列表。这样“我的牌局”和“全部牌桌”不再复用同一个语义冲突的路由。

详情采用同一横屏内容区内的导航，不改变侧边栏当前选择；返回只回到保留原筛选条件和滚动位置的存档列表。

## 5. 数据与模块边界

### 5.1 唯一数据源

`AppSession` 继续持有应用内唯一 `LocalPokerStore`。存档页面不得新建第二个 store、直接打开 JSON 文件或缓存另一份可写聚合状态。

`AppSession` 提供应用层存档操作：

- 加载全部或筛选后的已完成记录。
- 选择并展示一条记录。
- 删除单局。
- 二次确认后删除全部。
- 在删除成功或新结算完成后刷新可观察状态。
- 把领域错误映射为中文页面错误。

### 5.2 展示模型

新增独立功能目录 `RiverClub/Features/History`：

- `HandHistoryView`：列表、筛选、状态和删除确认。
- `HandHistoryDetailView`：单局最终结果。
- `HandHistoryPresentation`：把完成记录转换为只读 UI 模型。
- `HandHistoryViewState`：加载、已加载、失败、筛选和选中记录状态。

展示模型只保存页面所需的值类型，不保存 `LocalPokerStore`、协调器、完整恢复快照或任何进行中观察。

### 5.3 冻结显示元数据

现有 `StoredHandRecord` 已永久保存牌桌 ID、盲注、完整最终牌面、底池、赢家、起始/最终筹码和净变化，但没有冻结牌桌名称与座位显示名。

本期为新完成记录增加可持久化的安全显示元数据：

- `tableDisplayName`。
- `seatDisplayNames: [SeatID: String]`。

这些字段只包含公开显示文本，不包含机器人模型、设置、牌堆、种子、检查点或决策内部数据。协调器在结算时使用当前冻结的 `TableSeatProfile` 和已选择牌桌摘要生成元数据，并与完成记录在同一事务中保存。

向后兼容规则：

- 旧存档没有新增字段时必须正常解码。
- 旧存档牌桌名降级为 `牌桌 <tableID>`。
- 旧存档玩家名降级为 `座位 <1...9>`。
- 不能为了补名称改写旧记录或静默重置聚合存档。

### 5.4 筛选

筛选条件可以组合：

- 日期：全部、今天、最近 7 个本地自然日、自定义单日。
- 牌桌：全部或某个已出现在存档中的牌桌。

存储查询扩展为安全的日期范围和牌桌过滤，只返回 `StoredHandRecord`。日期范围按记录保存的 `LocalDay` 比较，不使用 `Date` 跨时区重新推导历史日期。结果始终按 `endedAt` 降序；相同结束时间使用稳定的手牌编号和标识排序。

首版不做分页。页面状态只保存筛选后的记录，唯一 store 仍是事实来源。

## 6. 删除语义

### 6.1 单局删除

单局删除需要二次确认，确认文案包含牌桌、日期和手牌编号。确认后调用 `LocalPokerStore.deleteHand(id:)`。

成功后：

- 从当前列表移除该记录并重新执行当前筛选。
- 若正在显示被删除记录，则返回列表。
- 不改变余额、账本、累计统计、当前会话、命令回执和永久身份保留集合。

失败后：

- 保留原列表和详情。
- 显示中文错误和重试入口。
- 不允许先乐观删除再尝试恢复。

### 6.2 全部清空

“清空全部存档”位于页面右上角菜单，只有存在记录时可用。二次确认后使用显式 `DeleteAllConfirmation.confirmed` 调用存储。

清空只删除记录索引和记录正文，不修改余额、账本、统计、当前会话、命令回执或已经使用过的手牌/结算/会话身份。完成后显示无存档空状态。

## 7. 页面状态与错误处理

页面状态包含：

- 加载中：横屏骨架列表。
- 已加载：列表或详情。
- 无存档：`完成一局后会在这里保存牌局记录`。
- 筛选无结果：`当前筛选条件下没有牌局`，提供清除筛选。
- 读取失败：显示错误和重试，不重置账户或覆盖存档文件。
- 删除失败：保留当前内容并提供重试。

每次进入“我的牌局”时重新读取唯一 store；从牌桌完成一局后返回该页面时也重新读取，因此不需要常驻双向缓存。

所有删除任务需要视图生命周期句柄。视图离开时可以取消尚未进入存储事务的任务；已经进入同步原子事务的删除以存储结果为准，随后通过重新读取对齐页面状态。

## 8. 安全约束

- `CompletedHandRecord` 继续作为唯一包含所有实际获发底牌玩家最终底牌的公开类型。
- `handRecords` 只返回已经成功提交的 `StoredHandRecord`。
- 不新增公开牌堆、随机种子、完整检查点、任意玩家观察或待结算摊牌接口。
- 正常牌桌继续只使用 `PlayerObservation`、`SpectatorObservation` 和安全摊牌投影。
- 存档展示模型不得被机器人决策服务引用。
- 删除记录后，手牌、结算、会话和业务编号仍不能重用。
- 错误文本和日志不得包含恢复快照或隐藏牌面。

## 9. 横屏 UI 规则

- 目标设备为 iPhone 16 Pro Max 全程横屏。
- 自动化使用本机可用的最新 Pro Max 模拟器，并保留 iPhone 16 Pro Max 真机视觉验收。
- 列表采用左侧筛选区加右侧记录区，系统安全区内不遮挡侧边栏。
- 详情使用九座位结果网格；头像和牌面不能被压扁，底牌始终完整显示。
- 删除确认使用居中弹窗，危险操作颜色与文字共同表达，触控区域不小于 44 点。
- 减少动态效果开启时取消列表和详情转场动画，但不改变内容或操作顺序。

## 10. 测试与验收

### 10.1 PokerSession

- 完成并保存的牌局可按日期范围和牌桌组合查询，结果稳定倒序。
- 未完成、等待结算或保存失败的牌局不出现在查询中。
- 新记录冻结牌桌名与九座位显示名；旧记录缺少元数据仍能解码并降级展示。
- 记录包含所有获发底牌玩家的最终牌，包括弃牌者。
- 单局删除和全部删除不改变余额、账本、统计、当前会话、回执和永久身份。
- 删除写入失败不产生部分修改，重试结果正确。
- 损坏存档不能静默重置账户。

### 10.2 安全边界

- 普通游戏和机器人继续无法访问对手底牌、牌堆、种子和检查点。
- 未完成牌局不能构造 `StoredHandRecord`。
- 存档元数据不包含机器人设置或内部决策信息。

### 10.3 RiverClub

- 侧边栏“我的牌局”显示真实记录，不再显示模拟牌桌列表。
- 大厅“全部牌桌”仍能通过独立路由进入牌桌列表并买入。
- 默认排序、日期筛选、牌桌筛选和组合筛选正确。
- 列表与详情正确显示真人净变化、底池、公共牌和九座位最终牌。
- 单局删除与全部删除必须确认；取消确认不写存储。
- 加载、空数据、筛选无结果、读取失败和删除失败状态正确。

### 10.4 核心 UI 路径

```text
游客登录
→ 进入全部牌桌并真实买入
→ 完成并保存一局
→ 回到侧边栏“我的牌局”
→ 打开详情
→ 验证弃牌玩家底牌
→ 删除单局并确认
→ 验证列表更新且余额不变
```

验收命令必须包含：

- 完整 `PokerCore` Swift Package 测试。
- RiverClub 单元测试。
- 存档核心 UI 测试。
- 通用 iOS `build-for-testing`。
- `git diff --check` 和洁净工作区确认。

## 11. 完成标准

- 真实牌局存档列表和最终结果详情可用。
- 日期与牌桌筛选可组合。
- 单局删除和全部删除安全、原子且需要二次确认。
- 已完成记录能显示包括弃牌者在内的所有最终底牌。
- 正常游戏和机器人隐藏信息边界无回归。
- 旧存档可向后兼容读取，不因新增显示元数据而重置。

- 全量自动化测试和通用 iOS 构建通过。
- 独立代码审查没有 Critical 或 Important。
