# River Club 普通桌会话、娱乐筹码账本与原子存档设计

日期：2026-07-14  
状态：方案 A 已确认，等待书面规格复核

## 1. 目标与边界

本子项目在现有 `PokerCore` 规则引擎之上，提供可独立测试的普通桌会话、娱乐筹码账本和本地原子存档能力，为后续机器人、锦标赛和 SwiftUI 协调器提供稳定接口。

本阶段包含：

- 普通桌买入、连续多手、结算后离桌和重新买入。
- 账户娱乐筹码账本、每日赠送与破产救济。
- 已完成牌局永久记录、会话快照、账本与统计的原子提交。
- 应用中断后的会话恢复、重复恢复幂等和损坏存档隔离。
- 使用现有 `CompletedHandRecord` 保存所有已获发底牌玩家的牌，包括弃牌玩家。

本阶段不包含：

- 单桌锦标赛、涨盲、排名和奖励。
- 机器人决策、行动倒计时和思考延迟。
- SwiftUI 页面接入、动画、音效或历史查看器。
- 云同步、跨设备存档、导出或真实货币功能。

## 2. 模块结构

在现有 `Packages/PokerCore` Swift Package 中新增独立 target 和 library product：`PokerSession`。该 target 依赖 `PokerCore`，但不依赖 SwiftUI、UIKit、数据库框架或系统单例。把两个模块放在同一个 Swift package 内，是为了使用 Swift 的 `package` 访问级别建立可信恢复边界；应用和机器人仍只能看到 `public` 安全接口。

### 2.1 EntertainmentChipLedger

职责：

- 保存账户当前娱乐筹码余额。
- 生成不可变账本流水。
- 校验并执行普通桌买入和离桌退回。
- 按设备自然日发放每日赠送。
- 按设备自然日和资格条件发放破产救济。
- 使用业务编号保证重复命令不会重复改变余额。

固定规则：

- 初始余额：128,500。
- 每日赠送：10,000，每个自然日最多一次。
- 破产救济：账户余额低于 2,000 且不存在未结算买入事务时，可补足至 20,000，每个自然日最多一次。
- 所有金额为不可购买、不可提现、不可兑换的本地娱乐筹码。

每条 `LedgerEntry` 保存：业务编号、发生时间、原因、变化前余额、变化量和变化后余额。变化量使用有符号整数，其余筹码值继续使用非负 `Chips`。

### 2.2 CashGameSession

职责：

- 保存牌桌标识、盲注、座位、玩家桌上筹码和当前手状态。
- 校验买入为 40–100 个大盲。
- 使用 `HoldemGame` 开始和推进每一手牌局。
- 一手完成后生成 `CompletedHandRecord`，进入等待提交状态。
- 只有原子提交成功后才能开始下一手。
- 只允许在没有进行中手牌时离桌，并把玩家桌上筹码通过账本退回账户。
- 真人玩家可在没有进行中手牌或待提交结算时重新买入；重新买入通过账本原子扣款，补充后的真人桌上筹码必须达到至少 40 个大盲且不超过 100 个大盲。

首个普通桌版本固定为九人桌：一个真人座位和八个受外部协调器驱动的座位。本阶段不实现机器人策略，但会话接口允许调用方按任意座位提交合法动作。

会话生命周期：

`未入座 → 已买入/等待开局 → 进行中 → 等待结算提交 → 已保存/等待下一手 → 已离桌`

非法生命周期转换必须返回明确错误，且不能部分修改会话或账本。

### 2.3 SessionRepository

职责：

- 原子保存一个 `PersistedAppState` 聚合根。
- 读取并验证最新完整版本。
- 保存永久 `StoredHandRecord`，支持按标识读取和删除。
- 隔离损坏快照，不覆盖最后一个可恢复版本。

`PersistedAppState` 同时包含：

- 规格版本。
- 当前普通桌会话快照。
- 娱乐筹码账本。
- 永久牌局记录索引和记录数据。
- 玩家累计统计。
- 已处理业务编号集合。

本阶段采用版本化 JSON 文件存储。写入流程为：同目录临时文件写入 → 数据同步 → 原子替换正式文件。任何步骤失败都保留旧正式文件。

文件路径由调用方注入；包内不读取全局 Documents 路径。后续 iOS 接入层负责选择受文件保护的 Application Support 目录。

## 3. 安全恢复快照

`PokerCore.HoldemGame` 为保护隐藏信息，不能直接编码或暴露内部牌堆。本阶段在 `PokerCore` 增加仅供同一 Swift package 内 `PokerSession` target 使用的可信恢复能力，但不能重新开放给普通 UI 或机器人。

实现采用独立 Swift Package 目标：

- `PokerCore`：继续只向外部公开安全观察、脱敏事件和完成记录；完整快照类型及 `HoldemGame` 的导出/恢复方法使用 `package` 访问级别。
- `PokerSession`：作为同一 Swift package 的独立 target，通过 `package` 级受控接口持有并保存完整进行中快照。
- SwiftUI 和未来机器人只依赖 `PokerSession` 暴露的安全会话接口，不取得完整恢复对象。

不得使用 `public Codable HoldemGame`、公开恢复快照、`@_spi` 或可被普通 `import PokerCore` 访问的完整状态接口。非 `@testable` 外部 API 编译探针必须证明这些恢复符号不可见。

恢复数据包含完整牌堆、所有底牌和规则状态，只用于本地自动恢复，不能出现在：

- `PlayerObservation`。
- `SpectatorObservation`。
- 脱敏事件。
- 进行中历史查询。
- 日志或错误文本。

恢复前必须重新执行牌张唯一、筹码守恒、座位、盲注、行动玩家和底池不变量校验。验证失败时返回“损坏会话”结果，账户账本和已完成历史仍可读取，但该会话不能继续。

## 4. 数据流与事务

### 4.1 买入

1. 调用方提供桌号、盲注、买入金额和唯一业务编号。
2. 会话验证买入范围及当前生命周期。
3. 账本验证余额和业务编号。
4. 在内存副本上同时创建会话与买入流水。
5. Repository 原子保存聚合状态。
6. 保存成功后才替换调用方当前状态；失败则会话和余额都保持原值。

### 4.2 每手结算

1. `HoldemGame` 进入完成态。
2. 会话生成不可变 `CompletedHandRecord`。
3. 生成永久记录、更新会话筹码和累计统计。
4. Repository 在一次原子写入中保存记录、会话、账本和统计。
5. 成功后允许开始下一手；失败时保留等待结算提交状态并允许使用相同事务编号重试。

同一手记录使用稳定的 `handID`。重复恢复或重复提交相同 `handID` 必须返回既有结果，不能重复添加历史或重复改变筹码。

### 4.3 离桌

1. 仅在没有进行中手牌及待提交结算时允许离桌。
2. 以唯一业务编号把玩家桌上筹码记入账户账本。
3. 原子保存新账本和已离桌会话。
4. 成功后清除可恢复的活跃普通桌会话，但永久牌局记录保留。

## 5. 时间与业务编号

包内不直接调用 `Date.now`、`Calendar.current` 或随机 UUID。

- `SessionClock` 提供当前时刻和设备自然日标识。
- 业务编号由调用方生成并显式传入每个幂等命令；重试必须复用同一编号。
- 测试使用固定时钟和固定编号。

自然日使用调用方注入的时区与日历生成 `YYYY-MM-DD` 业务日期。应用运行期间时区变化只影响下一次命令，不追溯已经入账的记录。

## 6. 统计

本阶段保存最小可验证统计：

- 已完成手数。
- 获胜手数。
- 普通桌累计投入。
- 普通桌累计净变化。
- 最大单手赢得筹码。

统计只在已完成牌局原子提交时更新，重复 `handID` 不重复累计。更详细的牌型、位置和对手统计留到机器人历史分析阶段。

## 7. 错误处理

使用可比较、可本地化映射的领域错误：

- 余额不足。
- 买入低于 40 个大盲或高于 100 个大盲。
- 重复或冲突业务编号。
- 当前生命周期不允许操作。
- 当前手未完成或结算尚未保存。
- 存档编码、写入、替换或读取失败。
- 存档版本不支持。
- 恢复快照损坏。

Repository 写入失败不能修改调用方持有的已提交聚合状态。重试必须复用原事务和业务编号。

## 8. 测试设计

### 8.1 账本测试

- 初始余额为 128,500。
- 买入扣款和离桌退回生成前后余额正确的流水。
- 相同业务编号重复执行不重复扣款或发放。
- 每日赠送同一天只能领取一次，跨自然日可以再次领取。
- 破产救济只在余额低于 2,000、没有未结算买入且当日未领取时补足至 20,000。
- 所有算术溢出和负余额路径均被拒绝。

### 8.2 普通桌会话测试

- 40 和 100 个大盲可买入，边界外被拒绝。
- 九人会话能够连续完成多手，庄位正确轮转。
- 进行中或等待结算提交时不能离桌或开始下一手。
- 结算失败后保持稳定状态，使用同一事务重试不会重复记录。
- 离桌后桌上筹码完整退回账户。

### 8.3 存档与恢复测试

- 模拟任意写入阶段失败时，旧正式文件仍可读取。
- 进行中牌局恢复到相同安全观察和合法动作。
- 损坏牌堆、重复牌、错误筹码和错误行动玩家无法恢复。
- 当前手未完成时，历史查询不返回该手任何隐藏底牌。
- 完成记录永久保存所有已获发底牌，包括弃牌者。
- 单局删除和二次确认后的全部删除不影响账本。
- 不支持的版本返回明确错误，不静默重置账户。

### 8.4 属性与集成测试

- 随机执行至少 100 个普通桌会话命令序列，余额与桌上筹码总量守恒。
- 相同初始存档和命令序列产生相同最终聚合状态。
- 重复提交全部幂等命令不会改变最终余额、记录数和统计。
- `PokerSession` 的公开观察不包含对手底牌、牌堆或随机种子。

## 9. 实施拆分

按以下顺序分别制定任务并进行审查：

1. `PokerSession` target/product、领域标识、配置常量和可注入时钟。
2. 娱乐筹码账本与幂等流水。
3. 普通桌生命周期和买入/离桌事务。
4. `PokerCore` 可信恢复边界与完整快照校验。
5. 文件型 Repository、版本化聚合状态和原子替换。
6. 每手完成记录、统计与幂等结算事务。
7. 连续多手、恢复和随机命令属性测试。
8. XcodeGen 接入与通用 iOS 构建验证。

每项先建立失败测试，再实现最小功能，提交信息全部使用中文，并在进入下一项前完成独立代码审查。
