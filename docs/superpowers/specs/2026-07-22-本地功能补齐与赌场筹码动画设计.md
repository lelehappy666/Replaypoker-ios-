# River Club 本地功能补齐与赌场筹码动画设计

## 1. 目标

在不引入服务器、多人联网、真金交易或云同步的前提下，补齐 River Club 当前可见但尚未实现的本地功能，并将普通桌的筹码移动优化为流畅、具有赌场游戏质感的动画。

本阶段完成后，用户可以：

- 在普通桌观看与实际筹码变化同步的分层筹码动画。
- 使用快捷聊天、表情和可即时生效的牌桌设置。
- 报名、取消、参加、恢复并完成本地单桌锦标赛。
- 编辑本地资料、选择内置头像或系统相册照片，查看成就。
- 通过本地密码和 Face ID 保护应用数据，并安全重置游客数据。

## 2. 范围边界

### 2.1 本阶段包含

1. 普通桌筹码动画和筹码音效优化。
2. 牌桌快捷聊天、表情气泡与机器人概率回应。
3. 牌桌设置面板和局内即时配置。
4. 本地单桌锦标赛的报名、扣款、退款、恢复、淘汰、排名与派奖。
5. 个人资料编辑、内置头像、相册头像与本地成就。
6. 本地启动密码、Face ID、修改密码与身份验证后数据重置。

### 2.2 本阶段不包含

- 实时多人通信、断线重连和服务端权威牌局。
- 生产级 Apple 账户、云端身份和跨设备同步。
- 真金充值、提现、兑换或任何现金价值能力。
- 多桌锦标赛、服务端调度和真实玩家匹配。
- 自由文字聊天、网络消息和生成式机器人对话。

## 3. 总体架构

实施分为三个可独立交付的子项目：

1. **牌桌体验层**：仅消费现有牌局事件和安全观察，不改变 `PokerCore` 规则真值。
2. **本地锦标赛层**：复用普通桌的规则、机器人决策和安全恢复边界，新增赛事生命周期、盲注结构与排名派奖。
3. **本地资料与安全层**：管理版本化的资料、成就、相册头像引用和安全偏好；敏感验证信息使用 Keychain，不写入普通 JSON 存档。

三个子项目共用现有娱乐筹码账本、牌局存档和机器人安全信息边界。每笔报名、退款和派奖都必须使用唯一业务标识保证幂等，不得因重试或重启重复改变余额。

## 4. 赌场筹码动画

### 4.1 视觉和时序

- 每次下注、跟注、加注、盲注、底池归集和派彩拆分为 5–7 个轻量筹码簇。
- 筹码簇按 35–55 毫秒间隔错峰启动，每簇使用略有差异的二次弧线、旋转角度和落点偏移。
- 飞行速度采用“快速离手—匀速滑行—轻微减速落桌”的时序曲线，落点仅允许一次小幅回弹。
- 多人派彩依照底池分层和每名赢家金额分别启动；同一赢家的同一笔奖励一次完成。
- “减少动态效果”开启时保留金额联动，将飞行缩短为简短位移与淡入淡出。

### 4.2 筹码堆联动

- 付出方座位筹码堆随各簇飞出逐层降低，接收方下注区、底池或赢家筹码堆随到达逐层增高。
- 动画期间的筹码堆是显示投影，最终金额始终来自规则状态，动画不能改变规则结果。
- 筹码数字、跟注数字和加注数字必须完整显示，不使用省略号；空间不足时优先缩小字号。

### 4.3 性能约束

- 移动中的筹码使用单次轻量绘制或缓存图层，不在每帧创建多层文字、阴影和 `GeometryReader` 子树。
- 同一事件只有一个时钟驱动动画进度，不叠加父子两层隐式动画。
- 新筹码事件到达时取消已失效的展示任务，不遗留飞行筹码或临时金额。

### 4.4 声音

- 筹码离手时播放一次轻声滑动，第一簇和最后一簇落点播放不同强度的筹码碰撞声。
- 连续筹码事件需要限制同类音效的最小时间间隔，避免叠加爆音。
- 用户关闭牌桌音效后立即停止已排队的声音。

## 5. 牌桌聊天、表情和设置

### 5.1 快捷聊天与表情

- 聊天面板提供预设中文短语，包括问候、称赞、思考和结束用语。
- 表情面板提供适合牌桌的静态符号，不读取系统键盘和网络资源。
- 用户发送后，内容在其座位附近的气泡中显示 2.5 秒，不覆盖手牌、筹码金额或状态标签。
- 每次用户发送后，最多一名未弃牌机器人按配置概率延迟 0.8–1.8 秒回应；同一局连续回应需有冷却时间。
- 机器人回应仅使用预设内容，不读取任何隐藏牌局信息。

### 5.2 牌桌设置

局内设置面板包含：

- 牌桌音效开关。
- 筹码动画开关与“慢、标准、快”三档速度。
- 当前牌型提示开关。
- 低于买入下限时自动补充娱乐筹码开关。
- 机器人难度、决策模型、风格和风险参数入口。

修改后立即保存；视觉设置立即生效，机器人设置从下一次机器人行动开始生效，不中断已在计算的当前决策。

## 6. 本地单桌锦标赛

### 6.1 生命周期

赛事状态为：可报名、已报名、进行中、已淘汰、待领奖和已结束。状态转换由本地赛事协调器串行执行，SwiftUI 不得直接修改赛事真值。

- 报名成功时原子扣除报名费并写入报名记录。
- 开赛前取消报名时原路退回全部娱乐筹码。
- 到达开赛时间后，已报名玩家可进入九人单桌，空位由本地机器人补齐。
- 应用重启时恢复赛事、盲注级别、座位筹码、淘汰顺序和当前牌局。
- 赛事结束后按名次派发奖励，记录个人统计与成就，并通过业务标识防止重复领奖。

### 6.2 牌局规则

- 采用九人德州扑克单桌淘汰制，行动顺序始终顺时针。
- 所有座位使用相同初始锦标赛筹码，锦标赛筹码不直接等同于账户余额。
- 盲注按固定级别和本地已运行时间递增，暂停或进入后台时不会在用户不知情的情况下跳过多个级别。
- 筹码归零且当前底池结算完成后淘汰；相同一手淘汰多人时按该手开始前筹码量排序确定名次。
- 机器人只消费安全公开观察，不能读取其他座位底牌、剩余牌堆或未来随机结果。

### 6.3 异常处理

- 余额不足、重复报名、名额已满、赛事已开始和数据不可恢复时显示明确中文错误。
- 扣款成功但报名存档失败时整体回滚，不保留半完成报名。
- 牌局或派奖保存失败时保留可重试状态，重试不重复扣款或发奖。

## 7. 个人资料与成就

### 7.1 资料编辑

- 昵称去除首尾空格后必须为 2–12 个可见字符，不接受纯空白内容。
- 用户可从现有内置真人头像库选择头像。
- 用户也可通过系统照片选择器导入图片，在本地裁剪为正方形并生成适合头像显示的尺寸。
- 相册权限被拒绝或图片解码失败时保留原头像，并提供继续使用内置头像的入口。

### 7.2 成就

首批成就覆盖：完成手数、获胜手数、特定牌型、连续获胜、锦标赛参赛、进入奖励名次和获得锦标赛冠军。

- 成就只根据已原子保存的普通桌记录和锦标赛结果计算。
- 每个成就显示名称、说明、当前进度、目标、锁定状态和首次解锁时间。
- 同一条业务记录不能重复推进成就。

## 8. 本地安全

- 启动密码使用系统 Keychain 保存带随机盐的单向校验值，不保存明文密码。
- Face ID 通过 `LocalAuthentication` 调用，只保存是否启用的偏好，不保存任何生物特征数据。
- Face ID 不可用、权限被拒绝或认证失败时，必须可以回退到本地密码。
- 连续认证失败仅阻止进入应用，不删除任何数据。
- 重置游客数据需要二次确认和已配置的身份验证；重置清除余额、牌局存档、赛事、成就、自定义头像和个人资料，然后恢复默认游客账户。

## 9. 数据与迁移

- 新增本地模型均携带 `schemaVersion`，使用现有原子文件写入方式。
- 旧存档第一次读取时补全默认资料、空赛事和空成就数据，不改变已有余额和牌局记录。
- 自定义头像存储在 Application Support 内的独立文件，存档只保存受控相对引用，不保存外部相册 URL。
- 密码校验值与生物识别凭据不进入普通存档、牌局历史或调试日志。

## 10. 验收标准

1. 筹码以 5–7 个错峰簇沿差异弧线飞行，落点有轻微回弹，筹码堆高度和金额同步变化。
2. 下注、底池归集、单人派彩和多人派彩都不留下悬空筹码，不改变最终规则金额。
3. 快捷聊天、表情、机器人概率回应和牌桌设置在 iPhone 16 Pro Max 横屏下不遮挡牌面、状态和操作区。
4. 本地锦标赛可从报名完整进行到派奖，重启后可恢复，重试不重复扣款或发奖。
5. 机器人在普通桌和锦标赛中均不能读取不属于自己的底牌、剩余牌堆和未来随机结果。
6. 昵称、内置头像、相册头像和成就均可持久保存，进程重启后显示一致。
7. 启动密码和 Face ID 能正确阻止未验证访问，认证失败不删除数据，重置必须二次确认并验证身份。
8. 旧版本现有余额、普通桌会话和完成牌局存档在迁移后不丢失。
