# River Club 普通桌 SwiftUI 可玩闭环设计

日期：2026-07-15  
状态：已确认
目标设备：iPhone 16 Pro Max，全程横屏

## 1. 目标

把现有静态九人桌 SwiftUI 原型、`PokerCore` 规则引擎、`PokerSession` 普通桌会话和 `PokerBot` 公平机器人系统接成第一版真实可玩闭环。

首版验收路径为：

`买入确认 → 自动发牌 → 真人与 8 个机器人依次行动 → 摊牌或提前结束 → 结算并保存 → 点击下一手`

SwiftUI 只负责展示安全状态和发送玩家意图，不判断扑克规则，不读取机器人底牌、牌堆、洗牌种子或恢复检查点。

## 2. 本阶段范围

### 2.1 包含

- 固定九人普通桌：一个真人座位、八个本地机器人座位。
- 买入确认后自动开始第一手。
- 真人完整合法操作：弃牌、过牌、跟注、下注、加注和全下。
- 30 秒真人行动倒计时；超时能过牌则过牌，否则弃牌。
- 八个机器人按规则行动顺序串行决策。
- 发底牌、公共牌、行动筹码和赢家高亮的基础逐步动画。
- 一手结束后原子保存完整牌局记录。
- 保存成功后显示“下一手”按钮，并使用最新设置开始下一手。
- 机器人筹码归零时，在下一手开始前自动补至 100 个大盲。

### 2.2 不包含

- 真人重新买入、完整离桌结算和离桌后重新入座。
- 冷启动后恢复进行中的牌局。
- 单桌锦标赛。
- 牌局记录查看器和机器人设置页面的完整 SwiftUI 界面。
- 聊天、表情、音效及高级筹码飞行动效。
- 实时多人联网。

应用进入后台时必须停止倒计时和机器人调度，不能在后台继续推进牌局；完整冷启动恢复在后续阶段实现。

## 3. 模块架构

在 `Packages/PokerCore` 中新增独立 target 和 library product：`PokerCoordinator`。

依赖方向为：

`PokerCoordinator → PokerCore + PokerSession + PokerBot`

`RiverClub → PokerCoordinator`

`PokerCore`、`PokerSession` 和 `PokerBot` 不反向依赖协调器或 SwiftUI。

采用独立模块而不是直接写在应用层，原因是协调器需要通过 Swift package 的 `package` 访问级别取得每个机器人自己的 `PlayerObservation`。该观察只在协调器内部转换为 `BotObservation`，不会成为 RiverClub 的公开接口。

### 3.1 应用级会话所有权

`AppSession` 在 Application Support 目录初始化唯一 `LocalPokerStore`，账户余额直接读取该存储，不再手工维护或直接执行 `chipBalance -= buyIn`。买入确认必须先通过 `LocalPokerStore.sitDown` 原子完成，成功后才创建协调器并进入牌桌；失败时保留买入面板和原余额。

`CashTableCoordinator` 接收并持有同一个 `LocalPokerStore` 引用，保证大厅余额、普通桌会话、每次动作检查点和最终记录来自同一真值来源。首版只处理本次运行中新建的会话；检测到需要冷启动恢复的旧活跃会话时显示明确错误，留给后续恢复阶段处理。

### 3.2 `CashTableCoordinator`

`CashTableCoordinator` 为 `@MainActor @Observable` 对象，依赖 Observation，但不依赖 SwiftUI。它持有：

- `LocalPokerStore`：唯一会话和持久化真值来源。
- `BotDecisionService`：异步机器人决策、超时、取消和旧版本拦截。
- 可注入时钟：真人倒计时、动画节奏和测试虚拟时间。
- 可注入标识与随机源：生成手牌编号、业务编号、洗牌种子和决策种子。
- 当前 `stateVersion`、动画队列和本手冻结的机器人设置。

公开接口只包含：

- 读取安全 `TableViewState`。
- 使用 RiverClub 传入的冻结 `BotSettings` 启动第一手或下一手。
- 提交真人操作意图。
- 通知进入后台或恢复前台。
- 重试失败的结算保存。

### 3.3 安全状态模型

`TableViewState` 只保存 UI 必需数据：

- 公开座位：座位号、显示名、头像标识、筹码、庄位、弃牌、全下和行动状态。
- 真人私有信息：仅真人两张底牌。
- 公共牌、当前底池、边池摘要和当前行动者。
- 真人合法操作和下注范围。
- 倒计时剩余秒数。
- 当前流程阶段、动画事件、赢家展示和错误面板。

禁止进入 `TableViewState` 的数据：

- 机器人底牌。
- 未发牌和完整牌堆。
- 洗牌种子、机器人决策种子。
- `HoldemCheckpoint` 或其他恢复对象。
- 尚未结束牌局的完整底牌映射。

## 4. 状态机

`TableFlowPhase` 定义以下阶段：

1. `preparingHand`：冻结设置、补足归零机器人、创建手牌。
2. `dealing`：播放两轮底牌动画。
3. `waitingForHuman`：显示真人合法按钮并运行 30 秒倒计时。
4. `botThinking`：当前机器人思考，其他输入禁用。
5. `animatingAction`：展示行动和筹码变化，暂停下一行动调度。
6. `revealingBoard`：依次展示翻牌、转牌或河牌。
7. `settling`：展示赢家、底池分配和最终筹码。
8. `savingResult`：原子提交牌局记录。
9. `awaitingNextHand`：显示“下一手”按钮。
10. `saveFailed`：保留结算画面，只允许重试保存。
11. `suspended`：应用进入后台，取消计时和决策任务。

SwiftUI 不得直接修改阶段。协调器只在当前阶段允许的入口中接受意图。

## 5. 一手牌数据流

### 5.1 开始

买入成功进入牌桌后，协调器自动执行：

1. RiverClub 调用 `AppSession.freezeBotSettingsForNextHand()`，并把不可变设置快照传给协调器；`PokerCoordinator` 不依赖 AppSession。
2. 检查八个机器人座位；筹码为零的机器人补至 `100 × 当前大盲`。
3. 生成唯一 `HandID` 和仅供规则引擎使用的洗牌种子。
4. 调用会话开始手牌并持久化检查点。
5. 从规则事件生成底牌动画队列。

机器人补充的筹码只属于虚拟对手桌上资金，不进入真人娱乐筹码账本，不改变真人账户余额或统计。

### 5.2 行动循环

每次规则或会话状态变化后递增 `stateVersion`，然后读取当前行动者：

- 若为真人，发布合法动作和下注范围并启动倒计时。
- 若为机器人，通过 package 安全接口读取该座位自己的 `PlayerObservation`，构造 `BotObservation` 和 `BotDecisionRequest`。
- 机器人决定返回后，必须同时匹配 `handID`、`stateVersion` 和当前行动座位。
- 匹配成功的动作仍交给 `PokerCore` 验证；协调器不能绕过规则引擎修改状态。
- 动作执行成功后生成行动动画；动画队列完成后才调度下一座位。

下注轮完成时，协调器调用会话推进接口。翻牌、转牌和河牌根据规则事件逐张展示，然后继续行动循环。

### 5.3 结算和下一手

进入 `settlementPending` 后：

1. 根据标准规则展示仍在摊牌玩家的底牌；弃牌玩家底牌在正常牌桌中继续隐藏。
2. 播放主池、边池分配和赢家高亮。
3. 使用唯一业务编号调用 `commitPendingHand`。
4. 保存成功后进入 `awaitingNextHand`。
5. 用户点击“下一手”后重新冻结最新机器人设置并开始新手牌。

所有玩家最终底牌，包括弃牌玩家底牌，仍按既有规则写入永久存档，但只能在牌局记录页面查看。

为支持正常摊牌展示，`PokerSession` 新增仅供同一 Swift package 使用的安全摊牌接口。它只返回仍有资格争夺底池且按规则需要亮牌座位的底牌，不返回已弃牌座位底牌，也不暴露完整 `CompletedHandRecord`。完整底牌映射仍只在结算提交后进入永久存档。

## 6. 真人操作区

操作区保留三个动态主按钮：

- 左侧：弃牌。
- 中间：过牌；若需要补齐当前下注则显示“跟注 + 金额”。
- 右侧：下注或加注。

下注滑杆：

- 最小值来自 `minimumBet` 或 `minimumRaiseTo`。
- 最大值来自 `maximumRaiseTo`。
- 步长使用当前桌最小筹码单位；首版沿用整数娱乐筹码。
- 预设按钮只在计算后仍处于合法范围时可用。
- 滑到最大值且该金额等于全下金额时，右侧按钮标题显示“全下”。
- 没有合法下注或加注时隐藏滑杆和右侧按钮。

真人点击按钮时只发送意图。协调器使用最新合法动作重新构造 `PlayerAction`，因此旧 UI 状态不能提交非法金额。

## 7. 倒计时与机器人调度

### 7.1 真人倒计时

- 每次进入 `waitingForHuman` 从 30 秒开始。
- 动画、机器人回合和结算期间不计时。
- 真人成功行动后立即取消当前倒计时任务。
- 超时动作通过最新合法集合选择：能过牌则过牌，否则弃牌。
- 同一状态版本只能触发一次超时动作。

### 7.2 机器人调度

- 同一时间最多存在一个机器人决策任务。
- 每个机器人使用稳定座位身份，例如 `cash:<sessionID>:seat:<seatID>`。
- 使用本手冻结设置；本手中修改设置不会影响正在行动的机器人。
- 状态改变、进入后台或离开牌桌时取消决策任务。
- 超时或策略失败由 `BotDecisionService` 返回合法保底动作。
- 旧版本结果即使稍后完成也不得提交。

## 8. 动画设计

`TableAnimationEvent` 至少包含：

- `dealHoleCard(seat, cardVisibility)`。
- `showAction(seat, action, amount)`。
- `moveCommitmentToPot(seat, amount)`。
- `revealCommunityCard(card, index)`。
- `awardPot(seat, amount, potIndex)`。
- `highlightWinner(seats)`。

默认基础节奏：

- 底牌间隔 80 毫秒。
- 翻牌每张间隔 180 毫秒。
- 转牌和河牌各 220 毫秒。
- 行动反馈 250 毫秒。
- 底池分配与赢家高亮 650 毫秒。

协调器先执行一次规则转换，再串行发布由该转换产生的动画事件。动画未完成前不接受下一动作或启动下一机器人。开启“减少动态效果”时保留事件顺序，但将非必要等待压缩为零。

## 9. SwiftUI 接入

现有 `PokerTableView` 改为消费 `TableViewState`：

- `PokerSeatView` 使用真实座位筹码、状态和行动高亮。
- 中央公共牌和底池改为真实数据。
- 真人底牌固定显示在底部本人座位附近。
- `BetControlBar` 从合法动作构造按钮、滑杆和预设金额。
- 顶部显示桌名、盲注和账户余额。
- 结算后在右下操作区显示“下一手”。
- 保存失败时在牌桌内显示错误面板和“重试保存”。

原型中的聊天、表情和设置按钮保留外观，但本阶段不增加业务功能。

## 10. 错误处理

- 非法或过期真人动作：不改变会话，刷新最新状态和合法按钮。
- 机器人取消：不提交动作；若当前状态仍需该机器人行动，可重新调度。
- 机器人超时或策略失败：执行过牌优先、否则弃牌的保底动作。
- 规则或会话错误：停止调度，保留最后稳定画面并显示可重试错误。
- 结算保存失败：进入 `saveFailed`，禁止下一手；重试使用相同业务编号，保证幂等。
- 后台暂停：取消倒计时、机器人任务和动画等待，不在后台模拟动作。
- 同一次运行恢复前台：从 `LocalPokerStore` 重新构造安全展示状态、递增版本并仅重启当前真人倒计时或当前机器人任务；不补播后台前未完成的动画。
- 日志只记录公开状态版本、阶段、动作类型和错误码，不记录牌堆、种子或机器人底牌。

错误面板位于牌桌内部，不能用临时 Toast 代替关键错误，也不能替换整个应用根界面。

## 11. 测试与验收

### 11.1 协调器单元测试

- 买入后自动发牌并进入正确首位行动者。
- 八个机器人严格按规则行动顺序串行执行。
- 真人合法动作映射覆盖弃牌、过牌、跟注、下注、加注和全下。
- 30 秒超时能过牌则过牌，否则弃牌，且只执行一次。
- 机器人错误、取消和旧状态版本结果不会产生非法或重复动作。
- 动画事件顺序与规则事件顺序一致，动画期间不调度下一行动。
- 本手冻结机器人设置，修改只影响下一手。
- 机器人归零后在下一手前补至 100BB，不改变真人账本。
- 结算记录包含所有实际获发底牌玩家的最终底牌。
- 结算保存失败不能进入下一手，使用相同业务编号重试成功。

所有时间、标识和种子通过依赖注入固定，测试不使用真实等待。

### 11.2 公平边界测试

- RiverClub 普通导入无法取得机器人 `PlayerObservation`。
- `TableViewState` 没有牌堆、种子、检查点或机器人底牌字段。
- 进行中牌局的 UI 编码状态只包含真人底牌。
- 弃牌玩家底牌只存在于已完成永久存档。

### 11.3 SwiftUI 与构建验收

- 从买入进入牌桌，完成一手并点击进入下一手。
- 真人操作按钮随合法动作动态变化。
- 九个座位在 iPhone 16 Pro Max 横屏安全区域内完整显示。
- 发牌、公共牌、筹码和赢家动画按顺序展示。
- 通用 iOS 设备 `build-for-testing` 成功。
- 使用可用的最接近 iPhone Pro Max 模拟器执行单元和 UI 测试；最终在 iPhone 16 Pro Max 真机或对应模拟器补充视觉验收。

## 12. 完成标准

满足以下条件后，本阶段完成：

- 静态牌桌数据全部替换为真实规则与会话状态。
- 真人和八个机器人能完成至少一手并进入下一手。
- 所有最终动作均通过 `PokerCore` 合法性验证。
- 正常牌桌无法查看机器人或弃牌玩家隐藏底牌。
- 结算记录原子保存，失败不会开始下一手。
- 机器人设置只从下一手生效。
- 协调器、Package 全量测试和通用 iOS 测试构建全部通过。
