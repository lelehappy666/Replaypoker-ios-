# River Club GitHub README 设计规格

日期：2026-07-14  
状态：设计已确认，等待规格复核  
目标读者：中文客户与项目团队

## 1. 目标

为 River Club iOS 原型制作一份适合 GitHub 项目首页展示的中文 README，使访客无需阅读源码即可快速理解产品定位、界面完成度、核心流程和工程质量。

README 同时承担三项职责：

1. 用统一的品牌视觉建立专业的第一印象。
2. 用真实项目内容说明目前已经实现的范围。
3. 为开发者提供清晰、可验证的本地运行与测试指引。

## 2. 内容边界

README 只陈述当前仓库已经实现或明确规划的内容，不暗示已具备联网对战、真钱交易、生产级身份认证、扑克规则引擎或后端服务。

项目定位统一描述为：使用无现金价值虚拟筹码、面向 iPhone 横屏体验的 SwiftUI 德州扑克 UI 原型。

README 使用中文为主。Swift、SwiftUI、XCTest、XCUITest 等通用技术名称保留英文。首版不额外维护一份英文 README，避免内容重复和后续同步成本。

## 3. 视觉方向

采用已确认的 B 方案“产品案例展示式”：

- 主色为深墨绿、近黑绿和古金色，与现有 `RCTheme` 保持一致。
- 首屏同时展示 River Club 品牌、项目一句话定位、技术徽章和核心界面组合图。
- 组合图以九人牌桌为主视觉，游戏大厅、牌桌列表、锦标赛和个人中心作为四张辅助缩略图。
- 整体气质克制、成熟，避免霓虹、强渐变或过多徽章造成视觉噪声。
- README 正文使用 GitHub 原生 Markdown 排版，装饰主要集中在顶部视觉资源中。

## 4. 图片策略

图片统一放在 `docs/images/`，使用便于长期维护的英文小写文件名。

计划包含：

- `river-club-cover.png`：README 顶部宽幅封面，包含品牌、项目定位和界面组合。
- `river-club-table.png`：九人牌桌主界面展示。
- `river-club-screens.png`：大厅、列表、锦标赛和个人中心的四宫格展示。

优先使用 iPhone 16 Pro Max 横屏模拟器的真实应用截图。当前开发环境只有 Command Line Tools，无法直接运行 iOS Simulator；若实施阶段仍无法使用完整 Xcode，则根据现有 SwiftUI 源码和已确认的视觉规格制作静态项目展示图。静态图必须忠实反映现有界面，不标注为“实机截图”。

所有图片需要在 GitHub 的浅色和深色页面中保持足够对比度；单张图片应压缩到适合仓库浏览的大小，避免无意义的高分辨率资源。

## 5. README 信息架构

正文按以下顺序组织：

1. 顶部封面图。
2. 项目标题、一句话简介和 Swift / iOS / SwiftUI 徽章。
3. 项目简介，说明产品边界和横屏原型定位。
4. 界面展示，包含九人牌桌主图和四个核心页面组合图。
5. 核心功能，概括七个核心界面、九人桌、买入流程和加载/离线/重试状态。
6. 核心体验流程：登录 → 大厅 → 选桌 → 买入 → 九人牌桌。
7. 技术实现，说明 Swift 6、SwiftUI、Observation、Repository 抽象和本地模拟数据。
8. 项目结构，展示主要源码与测试目录，不罗列生成物。
9. 本地运行，列出 Xcode、iOS 和 XcodeGen 前置条件以及生成项目步骤。
10. 测试，列出单元测试与 UI 测试命令或 Xcode 操作方式。
11. 当前范围与后续方向，明确原型限制。
12. 免责声明，说明虚拟筹码无现金价值，项目不提供充值、提现或兑换。

## 6. 文案原则

- 语气专业、直接，避免营销夸张和无法验证的性能描述。
- 先讲用户能看到的结果，再讲实现方式。
- 每个章节保持短段落和紧凑列表，方便 GitHub 扫读。
- 统一使用“River Club”，不再用目录名 Replaypoker 作为产品名称。
- 不使用 Replay Poker 的商标、Logo、插画或专属视觉资产。
- 不出现真实货币符号、奖金兑现或收益暗示。

## 7. 工程信息准确性

README 中的功能清单以当前源码为准：登录、游戏大厅、牌桌列表、买入确认、九人牌桌、锦标赛和个人中心。

运行说明以 `project.yml` 为准：Swift 6、iOS 18+、iPhone 目标设备和横屏方向。README 不承诺仓库内不存在的依赖管理、CI、发布包或 App Store 状态。

测试说明以 `RiverClubTests` 和 `RiverClubUITests` 为准。实施阶段必须核对测试命令与当前 scheme；如果环境无法运行完整 Xcode，只能说明验证受环境限制，不得声称测试已通过。

## 8. 验收标准

- GitHub 首页首屏能清楚看到产品名称、iOS/SwiftUI 定位和主界面视觉。
- 中文读者在一分钟内可以理解项目目标、已实现页面和核心流程。
- 所有图片路径有效，桌面与移动宽度下均不会溢出。
- 功能、技术栈、项目结构与运行要求均可由仓库文件验证。
- README 明确区分已实现原型、未实现能力和后续方向。
- README 不包含真钱扑克暗示或第三方品牌侵权素材。
- Markdown 链接、锚点和代码块可正常渲染。

## 9. GitHub 发布方式

从 `main` 创建 `agent/add-project-readme` 分支。设计规格作为独立提交保存；README 与图片在实施完成并验证后形成后续提交。推送分支后创建 Draft PR，由用户在 GitHub 上最终确认和合并。

当前本地仓库尚未配置可见的 GitHub remote，且环境中没有 `gh` 命令。发布阶段需要先确认目标 GitHub 仓库地址，并确保可用的 GitHub 认证方式。
