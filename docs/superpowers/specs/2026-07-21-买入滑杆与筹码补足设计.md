# 买入滑杆与筹码补足设计

## 目标

- 修复点击“立即入桌”后，买入界面因滑杆区间为零而触发的 `max stride must be positive` 崩溃。
- 新账号初始娱乐筹码调整为 1,000,000。
- 现有账号若低于 1,000,000，仅执行一次补足，不删除现有牌局存档。

## 根因

`BuyInSheet` 在余额低于最低买入，或最高买入与最低买入相同时，仍创建上下限相同的 SwiftUI `Slider`。该控件内部无法用正数步长分割零长度区间，因此直接崩溃。

## 方案

### 买入界面

- 新增可滑动区间判定，只在上限严格大于下限时创建 `Slider`。
- 区间为零时显示同样高度和色彩的静态轨道，保持 UI 布局不跳动。
- 余额不足时继续禁用确认按钮，并显示明确错误文案。

### 娱乐筹码

- `SessionEconomy.initialBalance` 改为 1,000,000，用于新建数据。
- 账本新增“欢迎补足”原因和幂等业务标识。
- 启动时检查补足版本：当账户余额低于 1,000,000 时，追加差额至 1,000,000；高于或等于目标时不扣减。
- 同一补足版本重复执行时返回原收据，不会重复增加筹码。
- 若现有未结算现金桌会话存在，仍只调整账户余额，不改动桌上筹码和牌局存档。

## 错误处理

- 补足失败时不破坏原账本，保持当前余额。
- 买入范围计算继续遵循 40 至 100 个大盲注，不改动牌桌规则。

## 测试

- 余额低于最低买入时，滑杆区间被识别为不可滑动。
- 最高买入等于最低买入时，不创建零长度 `Slider`。
- 新账本初始余额为 1,000,000。
- 低余额旧账本一次补足至 1,000,000，重复调用余额不变。
- 原余额高于 1,000,000 时不扣减。
- 保留原有买入、结算和牌局存档测试。
